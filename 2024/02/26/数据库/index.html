2024/3/4 数据模型

数据模型的四个世界：
1. 现实世界
2. 概念世界:与具体的DBMS和计算机无关
i）几种模型： - 实体-联系（E-R）模型 基本概念：实体，属性，联系，以及它们之间的连接关系
实体：矩形,联系：菱形，属性：椭圆,带下划线的属性为各实体集的关键字属性
为了刻划实体间的函数对应关系，必须在线段边上用1:1（一对一），1:n（一对多），n:m（多对多）等注明
参考 - 扩充的实体-联系（EE-R）模型 - IS-A联系（单向箭头）
如果实体B是实体A的子集，则实体A是实体B的超集，实体B是实体A的子集
子集B可以通过IS-A联系继承超集A的所有属性 - 弱实体（依附某些实体存在）：向关系打单向箭头 - 面向对象模型：对象，类，消息 - 谓词模型 3. 信息世界：
以概念世界为基础，选用特定DBMS构造而成的逻辑数据模型
关系中的基本概念： - 关系模式：一个关系的关系名及其属性名的集合构成该关系的关系模式 - 关系数据库模式：该关系数据库中所有关系的关系模式的集合 - 元组：关系的每一行 - 关键字：能唯一标识关系中的一个元组，不含多余的属性值 - 主关键字 - 外关键字 4. 计算机世界：是DB的最终实现结构

2024/3/11 关系模型

  • 关系数据结构
    • 表结构
      • 表框架由n个命名的属性（attribute）组成，n被称为表的元数
      • 表框架每行数据被称为“元组” ，一个表框架可存放m个元组，m被称为表的基数（Cardinality）
    • 键：

      • 如果表A中的属性集F是表B的键，则称该属性集F为表A的‘外键’（或称‘外关键字’）||外键

      • 不能只根据一张表中已有的元组来判断其有哪些‘候选键’，必须根据表中属性的语义含义及其相互联系才能确定表中的‘键’的组成

    • 关系

    • 关系操纵与关系代数中的五种基本运算之间的对应关系

24/3/18 关系代数中的扩充运算

关系代数的扩充运算： - 交运算 - 除运算 - 自然连接运算

  • 交运算
    • 条件：同类关系
    • 结果：关系模式不变，由所有既属于关系R也属于关系S的元组所组成的集合
  • 除运算(详见P119)
    • 运算条件：Head(S)属于Head(R)
    • 可以假设为:
      Head(R)={A1,A2,……,An,B1,B2,……,Bm}
      Head(S)={B1,B2,……，Bm}
    • 其中:
      • S被称为‘除数关系’
      • R被称为‘被除数关系’
      • 结果关系被称为‘商’
  • 联接运算\(\Join\)
    • 基本条件:\(i\theta j\) 其中i是关系R中的属性，j是S中的属性
    • \(\theta\)是比较运算符
      • 大于，小于，等于联接

24/4/15 数据库安全

  • 数据库安全就是研究有关实体的主/客体划分以及主/客体之间的访问关系的控制

  • 主体访问客体需要进行一定的安全控制与检查，目前存在三种控制方式:

    • 身份标识与鉴别
      • 每个主体必须有一个标志自己身份的标识符(以区别不同的主体)以及一个用以验证其身份的访问口令
    • 自主访问控制（DAC）
      • 是一种基于存取矩阵的安全控制模型
      • 此模型由主体，客体，存/取操作三部分内容构成了一个矩阵
    • 强制访问控制
      • Bell-Lapadula模型:向上不可读,向下不可写，一种神奇的模型。（top机密可以改变但不会泄露-向下不可写，也不会被看到-向上不可读）

  • 数据完整性：防止非法使用插入，删除，修改等影响数据完整性的操作

    • 控制手段
      • 对存储数据错误的检测
      • 事务回卷功能
    • 三类数据完整性
      • 实体完整性
      • 关联完整性
      • 用户定义完整性约束
    • 隐蔽通道
    • 审计:跟踪记录用户对数据的访问操作 与审计有关的保护措施有:
      • 审计事件控制
      • 入侵检测和应对
      • 审计记录的保护

24/5/5 事务

由某个用户所执行的一个不能被打断的对数据库的操作序列被称为‘事务’
事务活动详见第五章PPT第14张
* 提交当前事务，事务在执行过程中对于数据库的所有修改操作都将永久地反应到数据库中，并且不可被取消

常见的并发执行错误: * 丢失修改 航班票的例子P31， 读到了同一个变量,导致有一个的修改被吞掉 * 脏读 P32,读到了未提交的修改结果,但是对方取消了 * 不可重复读 P33，一个事务的执行过程中，前后两次读同一个数据对象获得的值不一致

事务的级别: * READUNCOMMITTED：未提交读 * 在该方式下，当前事务不需要申请任何类型的封锁，可能会读到未提交的修改结果 * 禁止一个事务以该方式去执行对数据的‘写’操作，以避免与其它并发事务的‘写’冲突现象

  • READCOMMITTED：提交读
    • 在‘读’数据对象A之前需要先申请对数据对象A的‘共享性’封锁，在‘读’操作执行结束之后立即释放该封锁
    • 以避免读取到其它并发事务未提交的修改结果
  • READREPEATABLE：可重复读
    • 在‘读’数据对象A之前需要先申请对数据对象A的‘共享性’封锁，并将该封锁维持到当前事务的结束
    • 可以避免其它的并发事务对当前事务正在使用的数据对象的修改
  • SERIALIZABLE：可序列化(可串行化)
    • 并发事务以一种可串行化的调度策略实现其并发执行，以避免它们相互之间的干扰现象

数据库的物理组织

记录文件记录的具体是什么，为什么要索引？记录的是同类型的还是不同类型的？ 索引应该是类似指针的存在 文件记录组织的综述特别抽象，但是大概可以看作维护一个序列，之类的？

  • 文件记录组织
    • 堆文件组织:记录可以放在文件的任何位置上，一般以时间先后为序
    • 顺序文件组织:按照升序或者降序存储
    • 散列文件组织:记录中的某个项值通过散列函数求得的值作为记录的存储地址
    • 聚集文件组织
  • 索引技术与散列技术
    • 顺序文件
      • 按照某个属性(项)的取值进行排序而构成的数据文件被称为顺序文件
    • 索引文件
      • 建立索引文件的目的：以记录的特征值（通常是一个或多个字段的值）为输入，能够快速地找出具有该特征的记录
      • 稠密索引:PPT39,44,47
      • 稀疏索引:有一种分段分治的美感（块状数组？？？）PPT 50,51
      • 多级索引:索引套索引:从第二级索引开始建立的都是稀疏索引,有一种线段树建树的美感PPT57，58，59（二级索引可以放内存里都不用磁盘IO所以多级的最终次数是2）
      • 重复键值索引
        • 稠密索引的重复键值索引唯一
        • 由于相同键值的记录可能占用若干个磁盘块，稀疏索引的重复键值索引不唯一 PPT65，66
      • 非顺序文件中的索引技术 PPT69，70，有一种散列哈希表的美感
      • 多维索引 #### 关系数据库的规范化理论
  • 概述 PPT9，10，13，16
  • 函数依赖（FD:functional dependencies）:PPT18,判断不是的方法:取相等的决定因素，其对应的依赖因素不同则不是，进阶可以看后面的例子
  • Armstrong公理系统
    • 基本推理规则
      • 自反规则:如果Y是X的子集，则:X->Y
      • 增广规则:如果X->Y,则XZ->YZ,PPT40
      • 传递规则:如果X->Y,Y->Z,则:X->Z
    • 扩充推理规则

最小函数依赖集定义: * 如果函数依赖集F满足下列条件，则称F为一个极小函数依赖集，亦称为最小依赖集或最小覆盖 * F中任一函数依赖右部仅含有一个属性 * F中不存在这样的函数依赖X->A,使得F与F-{X->A}等价 * F中不存在这样的函数依赖X->A,X有真子集Z使得F-{X->A}U{Z->A}与F等价 传送门

范式:PPT70 * 第一范式:关系模式R(U)中的每个属性值都是一个不可分割的数据量 * 第二范式:设有关系模式R(U)是第一范式，且每个非主属性都完全函数依赖于关键字，则称关系模式R(U)满足第二范式，记作R属于2NF * 第三范式:如果关系模式是2NF，而且它的任意一个非键属性都不传递地依赖于任何候选键，则R属于3NF * BCNF:设关系模式R(U)满足1NF，且若X->Y时X必包含有该关系模式的关键字，则称关系模式R(U)满足BCNF范式,并记作R属于BCNF * 第四范式PPT124 * 多值依赖:PPT104,111,114 * 非平凡的多值依赖:PPT110 传送门
